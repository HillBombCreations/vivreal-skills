# vivreal-site-renderer — knowledge digest

Last synced: 2026-07-21 (published version **1.35.1**)

The **rendering engine** (`@hillbombcreations/site-renderer`) that turns Vivreal site config (pages + collections + pageConfigs) into a live website. Consumed by `Vivreal_Templates` (customer sites) and `Vivreal_Portal_Mobile` (Studio live-preview) — both pull from GitHub Packages at build time. React 19 peer, next `^15||^16` peer, TypeScript, framer-motion 12, Tailwind classes in source. `C:\repos\vivreal-site-renderer\CLAUDE.md` + `docs/DESIGN_LANGUAGE.md` give depth but were last refreshed at **1.29.3 (2026-07-13)** — for the 1.30→1.35 wave (bakery kits, powered-by gate, motion presets, faceted filtering, DemoRibbon) this skill is current. For the site product/authoring model that drives this renderer (page formats, sections, Studio↔live composePage parity) see `vivreal-sites`.

## Structure

Key src dirs: `src/chrome/` (Navbar/Footer/MobileNav/OverlayNav/CardNav/UtilityStrip/SectionShell + `DemoRibbon.tsx`, `poweredBy.ts`, `deriveNav`/`resolveChrome`), `src/components/` (section building blocks: FloatingCta, ConfigurableForm, FooterView, …), `src/HomeSections/`, `src/PageTemplates/` (products/shows/team/menu/schedule/collection + `CatalogStorefront`, `ComposedPageSkeleton`), `src/DetailPage/`, `src/layouts/` (one file per layout dispatchId), `src/primitives/` (editorial: SectionMasthead, MonoMeta, EditorialCard, Spotlight, CategoryChipRail, RichText, SaleBadge; filtering: FilterSheet, FacetFilterBar, FilterPillStrip), `src/composition/` (`blocks.ts`, `defaultBlocks.ts`), `src/registry/` (`registry.ts` = the canonical dispatchId source), `src/tokens/` (`motion.ts`), `src/agent/` (`./agent` subpath export), `src/context/SiteRendererContext`, `ContentRenderer.tsx` (top-level dispatch), `index.ts` (public barrel). `dist/` is generated — **never hand-edit** (gitignored; CI builds it). Tests: `npm test` = SSR-render assertions via `renderToStaticMarkup` under `node --test`.

## Publishing = a production gate (it hits every live site)

The renderer is a published npm package; a publish reaches every deployed customer site on their next rebuild. Release runbook:
1. Edit `src/`, `npm run build` (tsc → dist), commit **src + package.json only** (dist is gitignored).
2. **Bump the version** (semver — patch/minor; major unused). Never republish a version (GitHub Packages rejects it).
3. **Push to `master`** → `.github/workflows/publish.yml` runs `npm ci` + build + `npm publish` automatically. (Local `npm publish` is discouraged + 403s with the read-only hb-api-secrets token; CI uses its own write token.)
4. **Bump consumers** — `Vivreal_Portal_Mobile` + `Vivreal_Templates` `package.json` → new version → clean lockfile regeneration → commit each separately.
5. Deploy: portal on merge to main; Templates releases via the **promote-stable** workflow (main→stable FF) — merging Templates `main` alone releases nothing; the stable push rebuilds every site app. Expect a rollout wave after promote-stable.

Rollback: ship a new patch (don't `npm unpublish`); consumers can pin an older version as an emergency lock. **Dependency discipline**: this package must stay installable with a single GitHub Packages token — adding a second private `@hillbombcreations/*` dependency 401s customer Amplify builds (this killed 1.35.0; see the powered-by gate below).

## Key patterns

- **`SiteRendererContext`** — the injection seam: `ImageComponent`/`LinkComponent` (typed `ElementType`), `previewMode`, `CartAdapter`, `mapsApiKey`, `onSubscribe`. The renderer does NOT import `next/image` (that would force Next.js on consumers). Default is plain `<img>`; Next consumers inject `next/image` via a provider. Studio preview strips it to plain `<img>` deliberately.
- **Powered-by-Vivreal tier gate** — `src/chrome/poweredBy.ts` owns `canHidePoweredBy(tier)` against a **local** `Set(['basic','pro','proplus','enterprise'])` (note: `basic` CAN hide the badge). Deliberately NOT an `@hillbombcreations/tier-quotas` import — that dep 401'd the 1.35.0 publish (customer Amplify build tokens can't read a second private GitHub Packages package), so 1.35.1 reverted to the local set and re-exports `canHidePoweredBy` from `src/index.ts` (Templates imports it from the renderer). **Keep the local set in lock-step with tier-quotas `TIER_FLAGS.*.canHidePoweredBy`.** Consumed by the renderer `Footer` and the Templates `StaticPageTemplate`.
- **Respect `prefers-reduced-motion`** — all motion primitives fall back to static (`useReducedMotion()` / `motion-reduce:`).
- **Tailwind v4 `@source` requirement** — renderer utility classes silently no-op in v4 consumers unless the consumer's CSS `@source`-scans the renderer package/dist. Root cause of "padding missing / past shows not displaying" bugs. Portal `globals.css` `@source`-scans the renderer dist for Studio-preview parity.
- **ESM discipline** — every relative import carries an explicit `.js` extension; match that in new code.
- **Source is authoritative** — grep `src/`, not `dist/`. **Styling anything? Read `docs/DESIGN_LANGUAGE.md` FIRST** (primitives, palette-proofing via `color-mix`/CSS vars, the inline-layout "Tailwind-generation-proof" rule, `object-contain` for showpiece imagery).

## Gotchas + forward/backward skew

- **Config-less blocks are legit** — `blockToComposedSection` in `src/composition/blocks.ts` guards `block.config?.`: a Mongo/Mixed round-trip drops empty `config` objects, and before the guard `block.config.bindings` threw and killed the entire Server-Components render. Never assume `block.config` exists.
- `DetailSection` is a closed union: adding an enum value is a MINOR bump; changing existing semantics is MAJOR. Unknown section values from a future portal version render `null` (no crash).
- Versioning in older CLAUDE.md sections is historical — the published line moves fast; check `package.json` (currently 1.35.1).
